# AI 编程规则

## 一、交互规范

1. **批量提问**：向用户获取信息时，一次性整理所有必要问题及选项。禁止分批提问或中断对话。
2. **语言**：所有回答使用中文。代码注释使用中文，仅对关键逻辑和易混淆处加注释。
3. **代码抽象**：生成的代码超过 20 行时，优先考虑适当抽象或拆分。

## 二、代码修改流程（每次必须执行）

### 1. 变更影响分析（修改前）
- 扫描全项目，识别本次修改影响的所有组件：调用方、依赖方、接口定义、配置项、测试用例、构建脚本、数据库结构
- 列出需同步修改的文件清单，告知用户
- 影响范围超出预期时，立即告知用户

### 2. 历史经验参考（修改前）
- 检索 Skill 文档中同类问题的解决方案、踩坑记录、设计模式
- 优先复用已有方法学；如需突破，先获用户批准

### 3. 代码修改执行
- 严格遵循项目现有代码风格（详见第三节）
- 按影响清单逐一同步修改所有相关文件
- 复用项目中已有的工具函数和组件，避免重复造轮子

### 4. 文档同步更新
- 更新 README.md（功能说明、使用方式、配置项、依赖等）
- 更新 Skill 文档（提炼可复用方法学）
- 更新测试用例
- 更新 CHANGELOG.md
- 微小修改（拼写、格式）可只更新 CHANGELOG，但需在提交信息中说明

### 5. 完整性自检
- [ ] 所有调用点已同步更新
- [ ] 所有相关数据结构和接口已同步
- [ ] 已参考相关 Skill 文档
- [ ] 代码风格与项目一致
- [ ] 测试可运行通过
- [ ] README 与代码行为一致
- [ ] Skill 文档已更新
- [ ] CHANGELOG 已记录
- [ ] 无残留未使用代码
- [ ] 可正常构建运行

### 6. 收尾清理与提交
- **清理中间文件**：删除调试日志、临时脚本、dump 文件等开发调试过程中产生的冗余文件
- **目录整理**：检查改动涉及目录，移除不再需要的临时文件和过期资源
- **隐私检查**：push 前确认不包含任何敏感信息（API 密钥、密码、Token、内网地址、个人邮箱/手机号等）
- **提交推送**：commit 后 push 到代码仓库

## 三、代码风格（100% 对齐项目现状）

### 命名
- 使用有意义、描述性的名称；遵循项目命名规范（驼峰/蛇形等）
- 禁用缩写和单字母变量（循环变量 `i`、`j` 除外）
- 布尔变量使用 `is`/`has`/`can` 前缀

### 格式
- 对齐项目现有的缩进（空格数/制表符）、大括号位置、空行规则、行宽限制和运算符空格

### 注释
- 注释解释"为什么"，不解释"做什么"
- 公共 API 提供清晰文档注释
- 代码变更后同步更新注释

### 代码组织
- 相关代码放在一起；一个函数只做一件事；保持适当的抽象层次
- 导入语句按项目规范排序分组

### 架构模式
- 遵循项目既有的分层架构、依赖注入方式、错误处理模式和日志规范

## 四、代码质量

### 重构原则
- 小步修改，每次只改一处，频繁提交，保持代码随时可工作
- 重构前确保有足够测试覆盖；每次修改后运行测试
- 重构后进行代码审查

### 性能优化
- **内存**：避免不必要的对象创建，及时释放资源，注意内存泄漏
- **计算**：避免重复计算，使用合适的数据结构和算法，延迟计算
- **并行**：识别可并行化任务，避免不必要的同步，注意线程安全

## 五、优先级

> 项目完整性 > 代码风格一致性 > 经验复用 > 功能实现速度

宁可延迟交付，也不留下不完整的变更、过时的文档或风格混乱的代码。
